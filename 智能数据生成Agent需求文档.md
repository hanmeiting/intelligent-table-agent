# 智能数据生成 Agent 系统需求文档

## 1. 项目背景与目标
本项目旨在开发一个基于大语言模型的智能数据生成辅助系统。用户可以通过自然语言对话或上传图片的方式，快速生成符合特定业务结构的结构化数据。系统采用直观的“对话+实时预览”交互模式，极大提升数据录入和创建的效率。

## 2. 技术栈架构
*   **前端**: Vue 3 (Composition API) + Js + 适用UI组件库（如 Element Plus / Ant Design Vue）
*   **后端**: Node.js + NestJS
*   **AI 核心**: LangChain (Node.js 版本) + 适配的 LLM (需支持多模态和流式输出)
*   **数据库**: 本地数据库（MySQL / PostgreSQL / MongoDB 等，根据现有业务情况定）
*   **通信协议**: HTTP/HTTPS (RESTful API) + SSE (Server-Sent Events) 用于流式数据传输

## 3. 页面布局与核心交互设计
### 3.1 页面布局
*   **主视图 (默认状态)**: 满宽的数据表格（Table），展示已有数据。包含分页、筛选、以及一个**“智能生成/唤起AI”**的入口按钮。
*   **对话视图 (展开状态)**:
    *   点击入口按钮后，从页面左侧滑出或展开一个对话面板。
    *   **挤压式交互**: 对话面板展开时，不是浮动遮挡表格，而是将右侧的表格内容向右挤压（自适应宽度），确保用户可以同时看到对话过程和表格的实时变化。

### 3.2 交互流程
1.  **发起请求**: 用户在左侧对话框中输入自然语言描述，或上传相关图片（如单据、草图等），点击发送。
2.  **流式响应**: 
    *   前端界面展示“AI思考中”或Loading状态。
    *   后端 Agent 开始处理，并通过 SSE 向前端流式返回 JSON 格式的结构化数据。
3.  **实时渲染**: 
    *   前端接收到流式 JSON 数据块并进行解析。
    *   由于是结构化数据，前端可以在对话框内以代码块或卡片形式预览，同时也可考虑直接在右侧表格的最上方/底部新增一行处于“生成中”状态的数据记录，动态填充字段。
4.  **确认入库**: 数据生成完毕后，用户可核对生成的 JSON 数据，点击“确认/保存”，前端调用后端普通 API 将数据正式写入本地数据库，并刷新表格。

## 4. 功能模块详细说明

### 4.1 前端功能模块 (Vue 3)
*   **动态布局模块**: 实现左侧 Sidebar 与右侧 Table 的弹性联动（Flex 或 Grid 布局平滑过渡）。
*   **数据表格模块**: 支持基础的 CRUD 展现，以及针对 AI 生成中的临时数据行高亮显示。
*   **对话交互模块**:
    *   **消息列表**: 区分用户消息和 AI 消息。
    *   **输入区**: 支持多行文本输入（Textarea），支持粘贴图片、点击上传图片。
    *   **流式解析器**: 能够健壮地接收和拼接 SSE 流返回的部分 JSON 字符串，并在界面上呈现打字机效果或动态更新。
*   **状态管理**: 管理对话上下文状态、抽屉展开状态、表格数据源同步。

### 4.2 后端与 Agent 核心功能 (Node.js + LangChain)
*   **API 路由层**: 
    *   提供获取历史数据、常规增删改查的 RESTful 接口。
    *   提供文件上传/图片暂存接口。
    *   提供 AI 对话交互的流式接口（SSE 端点）。
*   **Agent 核心引擎与工具 (LangChain)**:
    *   **多模态处理**: 将用户上传的图片转换为模型可理解的消息格式。
    *   **Prompt 工程**: 定义系统预设词，明确告知大模型输出必须是严格的 JSON 格式，并定义 Schema。
    *   **Agent Tools (所需工具集)**: 对于您的 Demo，建议配置以下 2-3 个核心工具，以保证生成的精准度：
        1.  **数据库表结构读取工具 (`get_table_schema_tool`)**: 让 Agent 能够实时查询本地数据库中目标表的字段名称、类型和约束要求，以确保生成的 JSON 字段 100% 匹配真实数据库。
        2.  **相似数据查询工具 (`search_existing_records_tool`)**: 当用户说“生成一个和上次差不多的商品”时，Agent 可以调用此工具查询本地数据库的已有记录作为参考上下文，从而更聪明地生成数据。
        3.  **最终数据输出工具 (`submit_structured_data_tool`)**: 利用 LangChain 的 Tool Calling / Function Calling 能力，将最终确定的结构化数据通过该工具输出。这是确保 LLM 输出稳定 JSON 的最佳实践。
    *   **流式生成处理**: 监听 `submit_structured_data_tool` 工具的调用过程，将参数块（Chunk）流式推送到前端。
*   **数据库服务层**: 接收最终确认的数据，执行本地数据库的 Insert/Update 操作。

## 5. 数据流与接口设计 (示例)
### 5.1 AI 聊天流式接口
*   **路径**: `/api/chat/stream`
*   **方法**: `POST`
*   **参数**: `{ messages: [{ role: 'user', content: '...', image: '...' }] }`
*   **返回**: `Content-Type: text/event-stream`
*   **数据块(Chunk)示例**: 逐步返回字符串 `{"name": "...", "age": ...}`

### 5.2 数据保存接口
*   **路径**: `/api/data/save`
*   **方法**: `POST`
*   **参数**: 生成的完整 JSON 对象
*   **返回**: `{ code: 200, message: "success" }`

## 6. 开发难点与注意事项
1.  **JSON 格式的流式解析**: LLM 在流式输出 JSON 时，前端接收到的字符串经常是不完整的 JSON，直接 `JSON.parse` 会报错。前端需要使用专门的库（如 `best-effort-json-parser` 或 `partial-json` 等）来解析不完整的 JSON，从而实现表格或预览区的平滑渲染。
2.  **幻觉控制**: 必须在 Prompt 中通过 few-shot 或严格指令，保证大模型不输出多余的 Markdown 标记（如 ```json ），直接输出纯数据，减轻解析负担。
3.  **布局挤压动画**: 左侧面板展开时，避免右侧 Table 出现滚动条闪烁或列宽突变，需要良好的 CSS 动画和过渡设计。

## 7. 核心体验提升建议 (Bonus/最佳实践)
为了让这个 Demo 的交互效果更加惊艳，建议在实现时考虑以下几点：

1.  **Human-in-the-loop (人工校验与编辑)**：
    *   Agent 生成的 JSON 即使格式正确，业务逻辑上也可能存在偏差（幻觉）。
    *   **建议方案**：在右侧被挤压后的 Table 最上方，动态插入一行高亮的新数据（处于“待确认”状态）。该行的单元格应该支持**内联编辑 (Inline Edit)**，允许用户在点击“保存入库”前手动修正 Agent 生成的错漏字段。
2.  **局部字段的重生成 (Regenerate)**：
    *   如果整条记录只有某个字段生成得不满意（比如根据图片生成的商品描述有误），可以支持用户在对话框或表格单元格旁点击“重新生成该字段”，Agent 仅针对该字段进行二次处理，无需全量重跑。
3.  **图片多模态的容错处理**：
    *   在处理上传的图片时（如单据扫描件），如果图片模糊导致关键信息提取失败，Agent 的设定应当是**反问用户补充信息**，而不是生硬地生成默认值或报错。
4.  **流式打字机动画与高亮联动**：
    *   在左侧对话框渲染流式 JSON 代码块的同时，右侧表格中对应的新增行也应当做到字段级的光标闪烁/高亮。例如：当流式数据推送到 `price` 字段时，右侧表格里 `price` 对应的单元格短暂发光，视觉引导性更强。
